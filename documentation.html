1. Abstract

The “Cryma” application (Cryptocurrency “Smart” Manager) is an application that tracks, displays and provides temporal details (starting with the present time) about the cryptocurrency prices.

2. Introduction

Cryma is an application dedicated to cryptocurrency enthusiasts. The application offers users the chance to track a multitude of cryptocurrencies, to analyze the progress of each currency throughout the prism of a time chart. The user can choose from three different time intervals, these having 24 hours, 7 days or a year, past from the current moment. At the same time, the user can select several currencies, which he wants to view constantly, and save them for later, as a list of favorites.

3.1 Technologies

For the development of the app, we will use:

  1. React Javascript library to build the user interface. It is maintained by Facebook and a community of individual developers and companies. React can be used as a base in the development of single-page or mobile applications. Compared to other frontend technologies such as Angular or Vue.js that are considered frameworks, React is a library. React is only concerned with rendering data to the DOM, and so creating React applications usually requires the use of additional libraries for state management and routing. Redux and React Router are respective examples of such libraries.
  2. Material-ui React UI framework for a faster and easier development. Material-UI is one of the top user interface frameworks for React out there. It gives developers a multitude of options, from a large array of text components, to premade grids, flexboxes, buttons, different types of inputs, icons, and so on. It also gives developers the chance to create their own CSS themes and use them across their apps.
  3. Chart.js library, in order to create the charts displayed in the app. Chart.js is a free open-source JavaScript library for data visualization, which supports 8 chart types: bar, line, area, pie (doughnut), bubble, radar, polar, and scatter. It is maintained by the community and is the second most popular JS charting library on GitHub.
  4. Axios, a popular HTTP client that helps us create requests. Axios is a promise-based HTTP client that sports an easy-to-use API and can be used in the browser.

3.2 API

API is the acronym for Application Programming Interface, which is a software intermediary that allows two applications to talk to each other. To be more specific it’s an information gateway that allows the backends of software and services to communicate with one another. How does this work? An API is the messenger that delivers your request to the provider that you're requesting it from and then delivers the response back to you.

APIs work using ‘requests’ and ‘responses’. When an API requests information from a web application or web server, it will receive a response. The place that APIs send requests and where the resource lives, is called an endpoint.

The API that will be used for the application is CoinGecko.

CoinGecko provides a fundamental analysis of the crypto market. In addition to tracking price, volume and market capitalization, CoinGecko tracks community growth, open-source code development, major events and on-chain metrics. The objective of this API is to elevate crypto participants’ understanding of fundamental factors that drive the market. It currently tracks nearly 5968 tokens from more than 410 cryptocurrency exchanges.

In the development of the application we will use these 2 endpoints:

  • GET /coins/${id}/market_chart/id?vs_currency=usd&days=1&interval=daily -> Get historical market data including price, market cap, and 24h volume (granularity auto). It can take the following parameters: id (passes the coin id), vs_currency (The target currency of market data), days (Data up to number of days ago), interval (Optional) (Data interval).
  • GET /coins/markets?vs_currency=usd&ids=bitcoin -> Lists all supported coins price, market cap, volume, and market related data. It can take many more parameters, like category, order, page or price_change_percentage, but for this application only 2 will be used: vs_currency (The target currency of market data) and ids (The ids of the coin, comma separated cryptocurrency symbols).

This API offers many more endpoints, such as:

  • GET /coins/{id}/history - Get historical data (name, price, market, stats) at a given date for a coin.
  • GET /coins/{id}/market_chart/range - Get historical market data include price, market cap, and 24h volume within a range of timestamp.
  • GET /coins/{id}/status_updates - Get status updates for a given coin.
    • There are many others endpoints. For a more detailed description of the API can be found at this link. But for the first version of the application the chosen two endpoints are enough.

3.3 Paradigms

Since the creation of React, the library has gone through significant changes. Nowadays there are three different ways of writing React, either using class components, functional components or a combination of both, each having their ups and downs. But lately, more and more React developers have switched to the functional components type of development, using React Hooks, since it makes the development process easier than the class components way. More than that, using functional components de-bloats the app as a whole, which makes it way faster.

So, for the development of the app, we will use React Hooks, which uses the functional components approach. Hooks solve a wide variety of seemingly unconnected problems in React that were encountered since the inception of the React library. Before hooks, it was hard to reuse stateful logic between components, but now, you can extract stateful logic from a component so it can be tested independently and reused in other components. So, in shorter words, creating our custom hooks helps us abstract logic and reuse it in other components.

3.4 App Structure

Front-end

  • The frontend application will be created using the create-react-app toolchain, that will set up the development environment, so that we can use the latest JavaScript features, and also optimize the app for production.
  • The app will be developed as SPA (Single page application), having one big component, which usually is called App. From our App component, the rest of the application will stem, basically creating a tree of smaller components, as you can see in the image below.
  • As mentioned, we will be using Material-ui framework in order to create our components. To give our components a proper design we will make use of the styled components approach provided by this framework.
  • To access the CoinGecko API, we will create an API controller. This will define a few methods that return diverse information, ex: get the price fluctuation of a certain coin over a period of time (24 hours, 7 days or 1 year).
  • In order to abstract the logic of fetching all the coin related data we will create a hook and it will be used wherever it will be needed.

    • As of now, we don’t see any other structural or architectural need for the frontend of the application.

Back-end

As of now we see two approaches for the back-end:

  • Go with a ‘serverless’ approach, meaning we won’t create a dedicated server for the application and just use the services already created in the front-end.
  • Create a dedicated node.js server, using the express framework. This server will act as a proxy to the CoinGecko API and serve the data to the frontend app.
